# Test Workflow — 実行手順の詳細

SKILL の本文は `SKILL.md`。このファイルは必要時に読み込む参照層。

## 開発開始時の前提確認（必須）

テストコマンドはプロジェクトごとに異なる。
`CLAUDE.md` の技術スタック・開発環境・質問ドキュメントの回答を確認し、未確定の場合は Human に次を質問してから実行すること：

- ユニットテストの実行コマンド（例: pytest, npm test, go test など）
- E2E の有無と実行方法（シェル・Playwright・クラウド上の検証など）
- ローカル依存サービスの起動方法（docker compose の有無など）

## 1. ユニットテストの実行

`CLAUDE.md` またはプロジェクトの README、`package.json`、CI 設定で定義されたコマンドを実行する（例として Python の場合）：

```bash
pytest tests/unit/ -v
```

実行後に以下を確認：

- 全テストのパス/失敗数
- 失敗したテストのエラーメッセージ
- カバレッジ（必要な場合）

## 2. 失敗時の対応

### 軽微な失敗（自動修正可能）

- 仕様書の変更に追従するだけの修正
- 明らかなタイポ・構文エラー

→ 修正後にユーザーに報告する

### 複雑な失敗（要判断）

- 原因が複数考えられる
- 修正方法が複数ある
- 本番環境への影響がある

→ error-handling Skill に従う

## 3. E2E テストの実行

ローカル／ステージングのみを既定とする。
本番環境や本番相当アカウントへの E2E は、Human の明示的な許可とプロジェクトで定義された手順がある場合に限り実行する。

ローカルで E2E スクリプトがある場合の例（変数名・パスはプロジェクトに合わせて読み替える）：

```bash
docker compose up -d
ENV=local bash tests/e2e/test_api.sh
```

コマンドや環境変数名が不明なときは推測で本番向け変数を使わず、必ず Human または質問ドキュメントで確認する。

## 4. テスト完了の報告

以下の形式で報告すること：

```
## テスト結果

### ユニットテスト
  合計: X件 / 成功: X件 / 失敗: X件

### E2E テスト（実行した場合）
  合計: X件 / 成功: X件 / 失敗: X件

### 失敗したテスト（ある場合）
  - テスト名: エラー内容

### 次のアクション
  （問題がある場合の推奨アクション）
```
